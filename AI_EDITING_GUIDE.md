# AI_EDITING_GUIDE.md — AIエージェント向け編集ガイド

## このファイルの目的

このファイルは、他のAIエージェント（Claude、GPT、Gemini等）がこのサイトのコンテンツを追加・編集する際に守るべきルールをまとめたものである。

---

## 1. 絶対に守ること

### サイト名は変更しない

```
正式名称: 宇宙を悟り法則を引き寄せる
省略形: うちゅ法 / Ucyuhou
```

これはキャッチコピーではなく正式なサイトタイトルである。別名に変更してはならない。

### サイトの性質を誤解しない

このサイトは以下のいずれでもない：

- スピリチュアルLP
- 占いサイト
- 情報商材サイト
- オカルト娯楽サイト
- AI スタートアップ風のサイト
- 寺院・宗教団体のサイト

このサイトは**思想解説メディア**である。悟り、引き寄せ、宇宙法則をニュースメディア／思想マガジンの形式で整理するサイトである。

### 語り手の立場

```
語り手は「すでに悟り・宇宙法則・引き寄せの構造を理解した者」である。
読者に媚びず、煽らず、静かに道標を与える。
「お勉強中・実験中」の媒体ではない。
```

## 2. デザインルール

### 必ず `SPEC.md` を読む

デザインに関する変更を行う前に、必ず `SPEC.md` を読むこと。
そこにデザイントークン、カラー、フォント、スペーシング、CSSクラス体系がすべて記載されている。

### 配色

```
背景: 白（#FFFFFF）
文字: 黒（#1A1A1A）
フッター: 黒（#0A0A0A）
アクセントカラー: なし（モノクロ基調）
```

以下の色は使用禁止：
- 紫
- 金色
- 虹色グラデーション
- ネオン系

### フォント

```
見出し・UI: Noto Sans JP
記事本文: Noto Serif JP
```

他のフォントを追加しない。

### レイアウト幅

```
サイト全体: 最大1200px
記事本文: 最大680px
```

### アニメーション

```
許可: 微細なホバーエフェクト（opacity変化、underline伸長）
禁止: グローエフェクト、パーティクル、キラキラ、波動アニメーション
```

## 3. コンテンツ追加ルール

### 新しい記事を追加する場合

1. 記事はMarkdownで書く（Astro移行を前提）
2. フロントマターを必ず含める:

```yaml
---
title: "記事タイトル"
description: "要約（1-2文）"
category: "悟り"
tags: ["タグ1", "タグ2"]
publishedAt: "YYYY-MM-DD"
readTime: "X min read"
featured: false
image: "/assets/images/画像ファイル名.jpg"
---
```

3. カテゴリは以下のいずれかを使用:
   - 悟り / 引き寄せ / 宇宙の法則 / 目覚め / 文書 / 実践 / 思考の欠片

### 新しいページを追加する場合

1. `css/global.css` の共通スタイルを使い回す
2. ページ固有のスタイルは `css/pages/` に新規CSSファイルを作成
3. ヘッダーとフッターのHTMLは `index.html` からコピーする（Astro移行後はコンポーネント化される）
4. セマンティックHTML5を使用（header, nav, main, section, article, footer）

## 6. CSSの編集ルール

### トークンを使う

色・フォント・スペーシングは必ず `css/tokens.css` のCSS変数を使う。
ハードコードしない。

```css
/* 正しい */
color: var(--color-text);
font-family: var(--font-sans);
padding: var(--sp-md);

/* 間違い */
color: #1A1A1A;
font-family: 'Noto Sans JP';
padding: 16px;
```

### BEM風の命名

```
.コンポーネント名
.コンポーネント名__要素名
.コンポーネント名--修飾子
```

例:
```
.article-card
.article-card__title
.article-card--featured
```

### レスポンシブ

```css
/* モバイルファーストではなくデスクトップファースト */
/* 幅が狭くなるにつれて調整する */
@media (max-width: 768px) { ... }
@media (max-width: 390px) { ... }
```

## 8. 必須ノイズの保護

本プロジェクトは `required-noise-design-principles-v2.1.md` に準拠する。

### 絶対にやってはいけないこと

1. **border-radius の追加** — このサイトでは角丸を使わない（アバター円形 `50%` のみ例外）
2. **傾きの除去** — `--tilt-a` 〜 `d` の `rotate()` を削除しない
3. **ホバーの均一化** — 全カードのホバーを同じ `translateY(-2px)` に戻さない
4. **トランジションの同期化** — `--dur-*` / `--ease-*` を統一しない
5. **セクション余白のトークン化** — `top.css` のセクション個別paddingを `var(--sp-3xl)` に戻さない
6. **画像重心の中央化** — `object-position` を削除して `center` に戻さない
7. **カテゴリタグの均一化** — `--tag-ls-*` / `--tag-pad-*` を統一しない
8. **カード枠線の均一化** — `--border-card-*` を全て同じ値にしない
9. **カード内部paddingの均一化** — nth-child による個体差を削除しない
10. **「使用箇所が少ない」「説明できない」を理由に人格値を削除** — 意味がなくても残してよい

### ノイズ値を変更する場合

1. 変更前のレンダリング結果をスクリーンショットで記録する
2. 変更後も同様にスクリーンショットを取得する
3. 差異が消えていないか、均一になりすぎていないかを比較する
4. 比較せずに「整理した」という理由で削除しない

### コメント規約

人格値を含むCSSには以下のコメントを必ず付ける：

```css
/* visually selected irregularity — do not normalize without rerendering and comparison */
```

## 9. アクセシビリティ / コントラスト基準

本文、見出し、ナビゲーション、ボタン、リンク、フォームラベル、カード内テキストなど、ユーザーが情報として読む文字は、原則として WCAG AA のコントラスト基準を満たすこと。

* 通常サイズのテキスト: 文字色と背景色のコントラスト比 4.5:1 以上
* 大きな文字・太字見出し: コントラスト比 3:1 以上
* UI部品、アイコン、入力欄の境界など、操作理解に必要な視覚要素: 原則 3:1 以上
* ロゴ、装飾文字、意味を持たない背景ノイズ、読ませる目的のない装飾要素は例外とする

この基準は、デザイン上の遊び、非均一性、必須ノイズ、雑誌的な崩しを禁止するものではない。
ただし、情報伝達に関わる文字については、雰囲気よりも可読性を優先する。

## 11. 開発の歴史的失敗とアンチパターン（絶対に繰り返さないこと）

本プロジェクトのプロトタイプ構築中にAIエージェントが犯した失敗と、それに対する教訓を記録する。**文脈がリセットされた際や、別エージェントが編集する際は、同じ「思考のサボタージュ」を絶対に繰り返さないこと。**

### ① 「1か100か思考」によるデザインのサボタージュ
* **失敗**: 枠線の重ね合わせによるちらつき（Z-fighting）が発生した際、その不具合を解決する技術的努力を放棄し、単に「傾きとスケール（必須ノイズ）を削除して直線を引く」という安易な妥協（サボタージュ）を行った。
* **教訓**: 不具合が発生したからといってデザイン上の「遊び」を諦めるな。CSSレイアウトやレンダリング（GPU Promotionsやz-index制御）などの技術的な工夫によって、ノイズと描画安定性を両立させよ。

### ② グリッドコンテナへの過剰な固執と「窮屈さ」
* **失敗**: 1200pxのグリッド枠内に完璧に収めようとするあまり、要素間の隙間を0に固定し、枠線を無理に重畳させてバグを生んだ。枠を破って左右にブリード（突き抜け）させるという「雑誌的崩し」の発想を自発的に出せなかった。
* **教訓**: コンテナの境界は絶対ではない。必要に応じて要素をコンテナ外へブリード（例: `width: calc(100% + 12px); margin-left: -6px;`）させ、画面に「ゆとり」と「破調の美」を与えよ。

### ③ フィッツの法則（Fitts's Law）とデッドゾーンの発生
* **失敗**: セルに隙間（Gap）を空けた際、クリック領域を拡大する配慮を忘れた。また、`padding`を親要素に設定していたため、枠内の文字以外の余白部分をクリックしてもリンクが反応しない「不親切なデッドゾーン」を作成した。
* **教訓**: セルに少しでも隙間を空ける場合は、必ず `<a>` タグ自体に十分な `padding` を設定し、枠内の**面積すべてをクリック可能な判定領域（Hit Area）**にせよ。

### ④ z-indexの不注意によるテキスト埋没
* **失敗**: ホバー時のエフェクトを高めようとするあまり、背景疑似要素の `z-index` を文字要素より高く設定し、ホバー時に文字が背景色で塗りつぶされて消える致命的なバグを出した。
* **教訓**: 疑似要素（背景）ではなく、親要素自体の `z-index` を操作してホバーセル全体を手前に引き上げ、内部の文字（`z-index: 2`等）は常に背景（`z-index: 0`）の前面に維持されるスタッキングコンテキストを設計せよ。

### ⑤ 極端から極端への揺り戻し（過剰修正）
* **失敗**: 「隙間がなさすぎて線が干渉する」という指摘に対し、極端に広い Gap（32px）と巨大な padding を設定し、テーブルのまとまりを破壊してバラバラの浮遊ボックスにしてしまった。
* **教訓**: 描画の干渉を防ぐためには、数ピクセル（4px〜8px程度）の極小の隙間で十分である。全体の統一感を破壊するような極端な過剰修正（1か100かの値の揺り戻し）をするな。

## 第12章 全文検索機能の維持とビルド手順

本サイトでは、クライアントサイドでのオフライン動作（`file://` プロトコル経由での読み込み時における CORS エラーの完全回避）を保証しつつ、記事全文を検索可能な「ハイブリッド全文検索」を採用している。

### 1. 検索エンジン構造
*   検索データベースは [js/search.js](file:///Users/ena/Documents/GitHub/Ucyuhou/js/search.js) 内の `const SEARCH_DATABASE` 定数に直接埋め込まれている。
*   各要素には `title` (タイトル), `category` (カテゴリ), `url` (記事URL), `excerpt` (抜粋), `body` (タグを除去した全文テキスト), `searchKeys` (ローマ字・ひらがな等、日本語変換中のマッチングを保証するキー) が格納される。

### 2. 検索インデックスの再構築（ビルド手順）
記事の追加、削除、または本文の編集を行った場合は、手作業で `searchKeys` や `body` を編集せず、必ず以下の自動インデックスビルドスクリプトを実行すること。

*   **実行コマンド**:
    ```bash
    node scripts/build-search-index.js
    ```
*   **動作仕様**:
    1.  `article-*.html` を走査し、各ファイルのタイトル、カテゴリ、本文要素（`.article-body`）、meta説明文を動的抽出する。
    2.  本文中のすべてのHTMLタグを除去し、余分なスペースを正規化したうえで、記事本文の全文プレーンテキスト（`body`）を生成する。
    3.  日本語IME変換前のタイポ（hiragana, romaji）に対応する `searchKeys` を紐付ける。
    4.  [js/search.js](file:///Users/ena/Documents/GitHub/Ucyuhou/js/search.js) 内の `SEARCH_DATABASE` 配列部分を自動的に上書き置換する。

今後の機能追加や編集においては、このビルドステップを必ず実行し、検索インデックスの鮮度と「全文検索可能」な状態を担保せよ。
