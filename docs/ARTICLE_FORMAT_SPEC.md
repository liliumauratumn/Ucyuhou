# ARTICLE_FORMAT_SPEC.md — 記事ファイル仕様

## 1. 保存場所とファイル名

公開記事は次へ置く。

```text
src/content/articles/<slug>.md
```

- `slug` は半角英小文字、数字、ハイフンのみ。
- 既存slugを再利用しない。
- 一時原稿や未校閲原稿を公開コレクションへ置かない。
- 新規記事は原則Markdownとし、MDXが必要な機能を勝手に追加しない。

## 2. frontmatterの標準形

```yaml
---
slug: example-slug
title: "記事タイトル"
subtitle: "タイトルを言い換えず、記事の問いまたは射程を示す一文。"
description: "一覧と検索結果に出ても意味が通る、記事内容の要約。"
category: "悟り"
categorySlug: "satori"
publishedAt: "2026-07-10"
publishedLabel: "2026年7月10日"
heroImage: "/assets/images/example-slug.jpg"
heroAlt: "画像に実際に写っている内容"
theme: "default"
tags: ["悟り", "非二元"]
authorName: "宇宙を悟り法則を引き寄せる"
readTime: "5 min read"
---
```

### 移行記事だけに残す項目

```yaml
sourceFile: "article-example.html"
```

`sourceFile` は旧HTMLから移行した記事の来歴である。Astro移行後に新規執筆した記事では省略する。存在しないHTML名を作らない。

## 3. 必須項目

| 項目 | 規則 |
|---|---|
| `slug` | ファイル名と一致させる。 |
| `title` | 一記事一問。誇張、煽り、結論の過剰な先取りを避ける。 |
| `subtitle` | タイトルの単純な言い換えにしない。 |
| `description` | 70〜140字程度を目安とし、一覧だけで内容が分かる文にする。字数を満たすために水増ししない。 |
| `category` | 8カテゴリーのいずれか。 |
| `categorySlug` | `CATEGORY_EDITORIAL_GUIDE.md` の対応表と一致させる。 |
| `publishedAt` | `YYYY-MM-DD`。Asia/Tokyo基準。 |
| `publishedLabel` | `publishedAt` と同じ日を日本語表記する。 |
| `heroImage` | `public` を除いた `/assets/images/...` 形式。実在ファイルを指す。 |
| `heroAlt` | 雰囲気ではなく画像内容を短く説明する。タイトルのコピーにしない。 |
| `theme` | 通常は `default`。宇宙通信だけ `cosmic`。 |
| `authorName` | 常に正式な著者名。 |

`tags` と `readTime` は任意だが、付ける場合は実内容に基づく。

## 4. readTimeの扱い

架空の飾り数字にしない。本文から概算する。

```text
表示本文の日本語文字数 ÷ 500 = 読了分数
小数点以下切り上げ
```

コード、frontmatter、HTMLタグは数えない。短文記事で読了時間表示が有益でない場合は省略できる。

## 5. 本文の基本形

通常記事は次を目安とする。

```markdown
<p class="dropcap-wrap"><span class="dropcap">最初の一字</span>導入本文...</p>

問いを成立させる導入。

## 第一の論点

根拠、教説、比較、具体例。

## 第二の論点

反論、限界、別の読み。

結論。本文の単純な再要約ではなく、最初の問いに答える。
```

- 冒頭のドロップキャップは既存デザインへ合わせる場合だけ使う。
- 見出しは本文の論理を進める。SEO語句を並べるために置かない。
- 1段落を一文だけで連続させない。
- 箇条書きは手順、比較、条件整理など、列挙する理由があるときだけ使う。
- 新規記事でインラインCSSを追加しない。
- 既存レイアウトやコンポーネントを本文から変更しない。
- 難しい言い回しやスピリチュアル系業界の専門用語を避けた、平易な文体（中学生にもわかるレベル、岩波少年文庫の読みやすさ）を徹底する。
  - 読者がスピリチュアル用語をあらかじめ知っている前提で書かない。
  - 専門用語をどうしても使用する場合は、初出時に誰にでもわかる言葉でその意味を解説する。
  - 「波動を上げる」「宇宙とつながる」「高次元」「覚醒する」などを説明なしで使わない。
  - 一文を長くしすぎず、漢語や難解な言い換えより、意味が直接伝わる日本語を優先する。
  - 業界用語を削ることで意味が不正確になる場合のみ、短い説明を添えて残す。


## 6. カテゴリー別の長さの目安

これはノルマではない。論点が完了した時点で終える。

| 種別 | 目安 |
|---|---|
| 悟り / 引き寄せ / 宇宙の法則 / 目覚め | 日本語本文 1,200〜3,000字程度 |
| 文書 | 日本語本文 1,800〜4,000字程度。資料の複雑さを優先する。 |
| 実践 | 日本語本文 900〜2,500字程度。手順と注意点が完結していること。 |
| 思考の欠片 | 日本語本文 300〜1,000字程度。無理に見出しを増やさない。 |
| 宇宙通信 | 日本語本文 600〜2,000字程度。通信形式の密度を優先する。 |

短い既存記事を長さだけのために膨らませない。改稿する場合は新しい論点か資料を追加する。

## 7. 引用と参考資料

### 直引用

- 正確な文言と出典を確認した場合だけ引用符またはblockquoteを使う。
- 翻訳した場合は「訳」または参照訳を示す。
- 出典不明の名言風文章を、人物の発言として掲載しない。

### 要約

人物や教説の考えを自分の言葉で要約する場合は、主語を明記する。

```text
トールは〜と述べる。
初期仏教の無我説では〜と説明される。
Ucyuhouではここを〜として読む。
```

### 参考資料欄

外部資料に依存する記事は、本文末に必要最小限の参考資料を置く。

```markdown
## 参考資料

- 著者名『書名』出版社、刊行年。
- “論文名,” 誌名, 年。
```

確認していない資料を体裁のために追加しない。

## 8. 宇宙通信の追加規則

- `category: "宇宙通信"`
- `categorySlug: "tsushin"`
- `theme: "cosmic"`
- 発信源は正式著者名。
- 公開日と通信内の地球時間を一致させる。
- 通常記事の断定や研究結果を通信形式で回避しない。
- 予言、医療判断、投資判断、政治的指示を送信内容にしない。

## 9. frontmatterで禁止する項目

実データと実装が存在しない限り、次を追加しない。

```text
reactionCount
commentCount
viewCount
likeCount
shareCount
ranking
verified
editorialStaff
```

## 10. 公開前の機械検証

```bash
pnpm exec astro check
pnpm build
```

ビルドが通っても、内容の正確さ、重複、画像権利、著者人格は別途編集校閲する。
