# PUBLICATION_WORKFLOW.md — 記事制作と公開前検証

## 1. 標準の役割順

```text
企画編集
→ 資料調査
→ 執筆
→ 画像ディレクション
→ 編集校閲
→ 必要最小の修正
→ Astro検証
→ ユーザー報告
```

一体型エージェントが全工程を実行する場合でも、頭の中で混ぜず、企画書、原稿、レビュー結果の順に成果を分ける。

## 2. Step 1：既存記事調査

企画前に `src/content/articles/` を走査する。

最低限、次を一覧化する。

- slug
- title
- category
- description
- tags
- 見出し
- 中心命題
- heroImage

新規企画と近い記事を最低3本挙げ、相違を説明する。近い記事がない場合は「該当なし」とする。無理に3本作らない。

## 3. Step 2：記事企画書

企画編集エージェントは次の形式で企画書を作る。

```markdown
# Article Brief

- 仮タイトル:
- category / categorySlug:
- 一文で示す問い:
- 一文で示す結論:
- 読者が読み終えた後に区別できること:
- 既存記事との差:
- 主な論点:
- 反論・限界:
- 必要な資料:
- 記述の確度区分:
- 画像の役割:
- 想定slug:
- 公開を止める条件:
```

企画書に本文を書き始めない。タイトルだけ変えた既存記事になっていないかをここで止める。

## 4. Step 3：資料調査

外部事実に依存する記事だけ調査する。すべての記事へ機械的に参考文献を付けない。

調査が必要な例：

- 人物の発言や著作
- 宗教・思想文書の成立史
- 翻訳の比較
- 科学、医学、心理学、量子力学
- 現在の制度、サービス、統計

調査時は一次資料または信頼できる専門資料を優先する。検索結果の要約だけで書かない。確認できなかった引用は削除する。

## 5. Step 4：執筆

- `EDITORIAL_CONSTITUTION.md`
- `AUTHOR_IDENTITY.md`
- `CATEGORY_EDITORIAL_GUIDE.md`
- `ARTICLE_FORMAT_SPEC.md`

を読み、企画書の範囲で原稿を作る。

原稿は校閲前に公開コレクションへ確定配置しない。エージェントの作業領域または差分上の一時原稿として保持し、レビュー合格後に `src/content/articles/<slug>.md` とする。

## 6. Step 5：画像

`IMAGE_EDITORIAL_POLICY.md` に従う。

- 既存画像再利用の理由
- 新規画像のファイル名
- 画像の出所または生成であること
- alt
- 既存画像との重複

を確認する。正式画像がなければ公開候補にしない。

## 7. Step 6：編集校閲

校閲は次の順で行う。

1. **企画適合**：最初の問いに答えているか。
2. **重複**：既存記事の言い換えではないか。
3. **事実**：人物、書名、年代、引用、科学的主張が正しいか。
4. **論理**：飛躍、自己矛盾、反論の無視がないか。
5. **著者人格**：語り手が別人格へ変わっていないか。
6. **カテゴリー**：記事の中心動作と分類が合うか。
7. **AI臭**：前置き、反復、機械的まとめ、水増しがないか。
8. **frontmatter**：値、日付、slug、画像、altが正しいか。
9. **画像**：重複、権利、内容不一致がないか。
10. **公開可能性**：捏造数字、未実装機能、プレースホルダーがないか。

重大な問題がある場合は `FAIL`、軽微な修正だけなら `PASS WITH FIXES`、そのまま公開候補なら `PASS` とする。

## 8. Step 7：修正

レビューで指摘された箇所だけを直す。レビューを理由に別記事へ作り替えたり、論点を増やしたりしない。

最大一回の通常修正で公開候補へ到達することを目標とする。根拠不足や企画重複が発覚した場合だけ、企画へ戻る。

## 9. Step 8：機械検証

プロジェクトルートで実行する。

```bash
pnpm exec astro check
pnpm build
```

続けて確認する。

```bash
git diff --check
git status --short
```

記事の追加・修正は `pnpm build` 内で検索インデックスへ反映される。`js/search.js` や生成済み `dist/` を手作業で編集しない。

## 10. Step 9：最終報告

最低限、次を報告する。

- 追加または変更した記事
- category / slug
- 既存記事との差
- 使用または作成した画像
- 参考資料の有無
- レビュー結果
- 実行した検証コマンドと結果
- 未解決事項

Git commit、push、デプロイは、ユーザーが明示的に依頼した場合だけ行う。

## 11. 公開停止条件

次のいずれかが残る場合、公開候補として合格させない。

- 出典不明の直引用
- 医療・科学・歴史についての重大な未確認断言
- 既存記事と中心命題が同じ
- 正式画像がない、または画像権利が不明
- 著者名、カテゴリー、日付、slugの不整合
- プレースホルダー、架空数字、未実装機能
- Astro checkまたはbuildの失敗

## 12. 日常運用向け一括簡略運用

通常の新規記事制作においては、日常運用を円滑にするため、以下の「一括簡略運用」を標準とする。

### 12.1 運用の流れ

ユーザーは「記事タイトルやテーマ、または短い思想メモ＋ここから記事を何本書くか」だけを提示する。
エージェントはこれを受け取り、**途中の承認待ちによる停止や、Article Brief、工程表のユーザーへの提示を行わずに**、以下の手順を一括して最後まで連続実行する。

1. **既存記事との重複確認**：既存記事と問いや結論の重複がないか走査する。
2. **テーマ・カテゴリーの選定**：複数の記事を執筆する場合は、同一内容の言い換えにせず、それぞれ独立した問いと結論を設計する。ユーザーの短いメモを見出しとして機械的に並べるのではなく、思想の核として解釈して適切なカテゴリーを決定する。
3. **本文執筆**：指定されたカテゴリーの文体と構造で執筆する。
4. **frontmatter作成**：仕様に沿って正確に作成する。
5. **画像生成**：`generate_image` を使用して、記事固有の画像を生成し配置する。
6. **編集校閲**：原稿の品質や整合性を校閲する。
7. **Astroへの追加**：`src/content/articles/` に記事を追加する。
8. **機械検証**：`pnpm build` および `git diff --check`、`git status --short` を実行する。

### 12.2 一括運用の途中停止（ブロック）条件

以下の重大な問題がある場合のみ、作業を途中で停止してユーザーに確認を求める。それ以外は最後まで一括で完了させる。

- 既存記事との重大な重複がある場合
- 必要な出典を確認できない場合
- 著作権上の問題がある場合
- 画像を正式に用意（生成）できない場合
- 新しいビルドエラーが発生した場合
- ユーザー本人の思想や体験を推測しなければ書けない場合

### 12.3 最終報告項目

一括完了後の報告は、以下の項目のみに簡潔にまとめる。

- 記事タイトル
- カテゴリー
- 追加した記事ファイル（リンク）
- 追加した画像ファイル
- build結果
- `git status --short` の結果
- （問題がある場合のみ）注意点

### 12.4 「思考の欠片」または「思考メモ」の例外ルール

- ユーザーが記事種別として「思考の欠片」または「思考メモ」と明示した場合は、必ず「思考の欠片」カテゴリー（`categorySlug: kakera`）に分類する。
- この場合、ユーザーから与えられた本文テキストは**一切加工・加筆・要約・再構成・言い換え・校正せず、そのまま反映**する。語順、改行、表記、句読点、誤字、脱字、不自然な空白なども「必須ノイズ」として維持する。
- 本文中へ画像、導入、見出し、解説、結論、まとめを勝手に追加してはならない。
- 難しい言い回しや理解不能な表現があっても、直しやふりがなを勝手につけてはならない。
- 逆に、明示的な指定がない場合は、文章が短いなどの理由であっても、自動的に「思考の欠片」カテゴリーへ分類してはならない。
