# 宇宙を悟り法則を引き寄せる (Ucyuhou) プロジェクト専用ルール

本プロジェクトでは、思想解説メディアに相応しい「静寂と知性」、および「雑誌的な崩し（必須ノイズ）」を両立するため、以下の実装制約を厳守すること。

## コーディングと問題解決の行動規範

### 1か100か思考の排除
実装した機能やデザイン（例: アニメーション、変形、装飾などのノイズ）が原因で描画不良やバグ（ちらつき、重なり順バグ、スタック崩れ）が発生した場合、その機能自体を削除して「解決」とすることを厳禁とする。
- デザイン要件や機能要求を「1」とし、不具合のない状態を「100」とするならば、その間を埋める技術的解決（例: レンダリングレイヤーの分離、トランジション制御、DOM構造の調整）を必ず模索せよ。
- 機能を消すことは「解決」ではなく「要件のサボタージュ」である。


## アクセシビリティ / コントラスト基準

本文、見出し、ナビゲーション、ボタン、リンク、フォームラベル、カード内テキストなど、ユーザーが情報として読む文字は、原則として WCAG AA のコントラスト基準を満たすこと。

* 通常サイズのテキスト: 文字色と背景色のコントラスト比 4.5:1 以上
* 大きな文字・太字見出し: コントラスト比 3:1 以上
* UI部品、アイコン、入力欄の境界など、操作理解に必要な視覚要素: 原則 3:1 以上
* ロゴ、装飾文字、意味を持たない背景ノイズ、読ませる目的のない装飾要素は例外とする

この基準は、デザイン上の遊び、非均一性、必須ノイズ、雑誌的な崩しを禁止するものではない。
ただし、情報伝達に関わる文字については、雰囲気よりも可読性を優先する。


## 操作性 / Fitts’s Law 基準

クリック、タップ、ドラッグ、選択など、ユーザーが操作するUI要素は、Fitts’s lawを考慮して設計すること。

* 主要ボタン、ナビゲーション、カードリンク、フォーム操作、閉じるボタン、保存ボタンなどは、十分なクリック・タップ領域を持たせる
* スマホ表示では、主要なタップ領域は原則 44px〜48px 四方以上を目安とする
* WCAG AA の最低基準として、ポインター操作対象は原則 24×24 CSS px 以上を下回らない
* 小さなアイコン単体を操作対象にする場合でも、見た目のアイコンサイズではなく、透明な余白を含めたヒット領域を確保する
* 隣接する操作要素同士は、誤タップを避けるため十分な間隔を取る
* 重要な操作は、画面の端・下部・親指が届きやすい位置など、操作しやすい場所に配置する
* 破壊的操作、削除、リセット、送信取消などは、主要操作と近づけすぎない
* インライン本文中の通常リンクや、装飾目的の非操作要素はこの限りではない

この基準は、デザイン上の遊び、非均一性、必須ノイズを禁止するものではない。
ただし、ユーザーが実際に押す・選ぶ・閉じる・送信する要素については、見た目の軽さよりも操作成功率を優先する。


## プロジェクト固有デザインルール

### 1. 角丸の絶対排除
- サイトの構造的要素（カード、ボタン、入力フォーム、画像など）に角丸（`border-radius`）を適用することは厳禁とする。アバター等の円形（50%）を除き、すべて `border-radius: 0` を徹底せよ。

### 2. グリッドの必須ノイズとFitts's Lawの両立
- グリッドアイテムやテーブル罫線に傾き（`rotate`）や伸縮（`scale`）の「ノイズ」を与える場合、隣接する枠線との衝突によるちらつき（Z-fighting）を防ぐため、Gap（隙間）を設定すること。
- ただし、テーブルレイアウトの塊としての美しさを損なわないよう、Gapは極小（4px〜8px程度）に維持すること。
- Gapを空ける場合、ユーザーの到達容易性（Fitts's Law）を担保するため、`padding` は親のコンテナセルではなく、必ず内部の `<a>` タグ自身に設定し、カード内の全面をクリック可能とすること。

### 3. ホバー変形時のレンダリング安定化
- 枠線やカードが傾いた状態でホバーアニメーションを行うと、アンチエイリアスの計算により線がちらつく。これを防ぐため、以下の記述を必須とする：
  - アニメーション対象の疑似要素等に `backface-visibility: hidden;`、`transform-style: preserve-3d;`、`will-change: transform;`、`translateZ(0)` を適用し、GPUレイヤーに昇格させる。
  - 親要素の `z-index` 降下にトランジション遅延（例: `transition: z-index 0s linear var(--dur-a);`）を設定し、マウスアウト時の重なり順の破綻を防ぐ。

## 記事制作タスクのルーティング

記事の企画、執筆、校閲、画像選定を行う場合、技術移行ルールだけで判断してはならない。必ず次を読むこと。

1. `docs/EDITORIAL_CONSTITUTION.md`
2. `docs/AUTHOR_IDENTITY.md`
3. `docs/CATEGORY_EDITORIAL_GUIDE.md`
4. `docs/ARTICLE_FORMAT_SPEC.md`
5. `docs/IMAGE_EDITORIAL_POLICY.md`
6. `docs/PUBLICATION_WORKFLOW.md`

役割に応じて以下のSkillを使用する。

- 企画: `.agents/skills/ucyuhou-article-planning/SKILL.md`
- 執筆: `.agents/skills/ucyuhou-article-writing/SKILL.md`
- 校閲: `.agents/skills/ucyuhou-editorial-review/SKILL.md`
- 画像: `.agents/skills/ucyuhou-image-direction/SKILL.md`

表示上の著者名と宇宙通信の発信源は、常に `宇宙を悟り法則を引き寄せる` に固定する。編集部、管理人、大いなる源泉、AIライターなどの別人格を作ってはならない。

## 検索機能の運用とビルド

本プロジェクトの正式な記事ソースは `src/content/articles/*.md` であり、検索インデックスはAstroビルド内で生成される。

- 記事の追加、削除、本文修正後は `pnpm build` を実行する。
- `js/search.js`、旧 `article-*.html`、生成済み `dist/` を記事更新のために手作業で編集しない。
- 公開前に `pnpm exec astro check` と `pnpm build` を実行する。
